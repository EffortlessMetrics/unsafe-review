# 0.3.4 Coverage-Instrument Usability Patch

`unsafe-review` provides baseline-aware unsafe evidence coverage artifacts for
repo badges, PR gates, line-comment plans, LSP/LLM context, and ub-review.

0.3.4 is the coverage-instrument usability patch. It publishes the
history-preserved source import of the post-0.3.3 coverage-slot and baseline
movement lane.

The release goal is narrow:

- introduce a slot-based unsafe-evidence coverage model that assigns each
  `unsafe` site a coverage slot and tracks which slots have evidence,
- add baseline recording and movement tracking with diff-scoped `no-new-debt`
  and `worsened_gaps` reporting,
- emit a baseline-aware badge that reflects coverage trend without implying
  proof,
- anchor comment plans to coverage gaps to improve comment-plan signal quality,
- add a `file:range` context packet for LLM and agent consumers,
- write an `unsafe-review-gate.json` routing manifest from every `first-pr`
  run,
- improve repo-scan diagnosability so consumers can distinguish scanned-but-
  empty from not-scanned,
- add stale JS-buffer span detection and stable-byte coverage v1 with span-as-
  arg detection and snapshot suppression,
- add candidate `new` and `lint` authoring subcommands,
- preserve the existing no-witness-execution, no-commenting, no-source-editing,
  and no-default-blocking boundaries.

## Highlights

- The SPEC-0029 unsafe-evidence coverage block assigns each `unsafe` site a
  coverage slot and tracks which slots have evidence, providing a structured
  baseline for gap reporting.
- The SPEC-0030 baseline movement keystone records baselines, tracks movement,
  and reports diff-scoped `no-new-debt` and `worsened_gaps`. The `baseline
  init` and `baseline add` subcommands create and extend baselines without
  changing advisory posture.
- The SPEC-0031 baseline-aware badge surfaces coverage trend in repo badge
  endpoints without implying proof or policy readiness.
- The SPEC-0032 comment-plan coverage-gap hardening anchors comment plans to
  coverage gaps, eliminating phantom anchors and improving signal quality.
- The SPEC-0033 file:range context packet gives LLM and agent consumers precise
  source span information in a structured JSON context packet.
- The SPEC-0034 `unsafe-review-gate.json` routing manifest lets downstream CI
  and agent consumers route decisions from a structured artifact without parsing
  human output.
- The SPEC-0035 repo-scan diagnosability adds scan-scope metadata so consumers
  can distinguish scanned-but-empty from not-scanned.
- Stale JS-buffer span detection flags use of a stale JS `ArrayBuffer`/
  `TypedArray` span after a GC-reentry point.
- Stable-byte coverage v1 adds span-as-arg detection (flags spans passed as
  arguments after reentry) and snapshot suppression (suppresses cards when a
  snapshot is taken before use).
- Candidate `new` and `lint` authoring subcommands support structured candidate
  creation and linting without changing the manual/advisory boundary.
- Opt-in `confirm --allow-heavy` runs emitted confirmation cues with explicit
  opt-in; `runtime_executed` is projected into output. Execution remains opt-in.

## Versioning

The public crates move together for this patch:

```text
unsafe-review-core 0.3.3 -> 0.3.4
unsafe-review-cli  0.3.3 -> 0.3.4
unsafe-review      0.3.3 -> 0.3.4
```

This follows the synchronized public crate release pattern used by earlier
patch releases.

## Preconditions

Before publication:

- source `main` must contain the import merge from this release-prep branch,
- the import merge must be byte-identical to swarm main `3805a0de`,
- no existing release tag is rewritten (`v0.3.3` is the latest published tag),
- no `0.3.4` crate is published before dry-runs and package inspection pass.

## Non-goals

0.3.4 is not:

- memory-safety proof,
- UB-free status,
- Miri-clean status,
- site-execution proof,
- calibrated precision/recall,
- policy readiness,
- witness execution,
- automatic commenting,
- automatic source editing,
- default blocking policy,
- automatic confirmation-cue execution (cues are emitted; `confirm
  --allow-heavy` is explicit opt-in only).

## Trust Boundary

`unsafe-review` 0.3.4 reports static unsafe-review evidence and baseline
coverage artifacts. It does not prove memory safety, UB-free status, Miri-clean
status, site execution, witness success, calibrated precision/recall, or policy
readiness.

It does not execute witnesses, post comments, edit source, or enforce blocking
policy by default.
