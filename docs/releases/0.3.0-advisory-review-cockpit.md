# 0.3.0 Advisory Review Cockpit Target

Date: 2026-05-31

`unsafe-review` 0.3.0 is the advisory review cockpit release. The goal is not
broader analyzer coverage, policy authority, live editor integration, automatic
comments, or witness execution. The goal is that a Rust maintainer can install
the tool, run one PR command, open one review kit, inspect the top
`ReviewCard`s, hand bounded card-sized work to a human or agent, and preserve
the trust boundary.

The release target is:

```text
cargo install unsafe-review --locked
unsafe-review doctor
unsafe-review first-pr --base origin/main
open target/unsafe-review/pr-summary.md
unsafe-review explain <card-id>
unsafe-review context <card-id> --json
unsafe-review support
```

## Product Boundary

0.3.0 reports static unsafe-review evidence gaps. It is not:

- memory-safety proof
- UB-free status
- Miri-clean status
- site-execution proof
- calibrated precision or recall
- policy readiness
- default policy gating
- automatic comments
- source edits
- witness execution
- agent execution
- live LSP or VS Code extension support
- Bun bug proof or Bun-specific release claim

Every user-facing surface must preserve the trust boundary: `unsafe-review`
finds unsafe Rust changes missing safety contract, guard, test, or witness
evidence. It routes reviewers to concrete next actions; it does not prove the
repository safe.

## Release Shape

The 0.3.0 user path centers on:

| Surface | 0.3.0 target |
|---|---|
| `doctor` | First-run readiness check that reports workspace signals, optional witness tools, and the trust boundary. |
| `first-pr` / `review` | One-command advisory review kit for PR review. Writes the manifest, cards, summaries, SARIF, comment plan, witness plan, receipt audit, saved LSP JSON, and repair queue. |
| `review-kit.json` | Manifest for the whole advisory review packet: emitted artifacts, summary counts, top-card handoff, and trust boundary. |
| `pr-summary.md` | Human reviewer cockpit: top card, operation, obligation, missing or weak evidence, next action, witness route, explain/context commands, and trust boundary. |
| `github-summary.md` | Bounded CI doorway that points to the full bundle without duplicating the report. |
| `comment-plan.json` | Artifact-only review budget with selected and not-selected reasons. Comments are planned, not posted. |
| `witness-plan.md` | Suggested witness routes and limits. It does not execute witnesses. |
| `receipt-audit.md` | Metadata-only audit of saved witness receipts against current ReviewCards. It does not run witnesses or make policy decisions. |
| `lsp.json` | Saved read-only editor projection. No live LSP claim. |
| `repair-queue.json` | Copy-only human/agent handoff buckets projected from ReviewCards. No agent execution, source edits, suppression, or repair-success claim. |
| `explain <card-id>` | Human-readable card explanation. |
| `context <card-id> --json` | Bounded card packet for humans or agents, including do-not-do boundaries. |
| Artifact verifier | Checks the first-pr review kit for parseability, artifact identity, card identity consistency, schema/trust-boundary wording, and absence of safety/proof overclaims. |

## Required Work

0.3.0 should be prepared by narrow source or swarm PRs:

1. Promote the full first-pr review kit to source:
   `review-kit.json`, `github-summary.md`, `receipt-audit.md`, and
   `repair-queue.json`.
2. Keep `pr-summary.md` as the reviewer cockpit front panel.
3. Keep `comment-plan.json` as review-budget metadata with selected and
   not-selected rationale.
4. Keep `context <card-id> --json` aligned with repair-queue buckets and
   do-not-do rules.
5. Promote only selected fixture-pinned analyzer/card-correctness controls
   needed for 0.3.0 dogfood or Bun investigation.
6. Prepare source version bumps, package lists, dry-runs, install smoke,
   first-pr smoke, artifact verifier smoke, explain/context smoke, doctor
   smoke, and support smoke.

Analyzer/evidence changes still start in `unsafe-review-swarm` by default.
Source receives curated promotions, release/public surface work, and urgent
hotfixes only.

## Release Cutline

Merge only release-confidence work before 0.3.0 if it is green and small:

- comment-plan review-budget tests
- API/domain/default-posture wording tests
- contract-text evidence tests
- artifact schema or verifier tests
- source-promotion fixes needed by the 0.3.0 review kit

Defer until after 0.3.0:

- fuzz harness expansion
- large scanner refactors
- large xtask module splits
- broad analyzer expansion
- policy-gate work
- live LSP/editor release work
- Bun-specific vulnerability claims

If a PR does not directly improve `first-pr`, `pr-summary.md`,
`github-summary.md`, `comment-plan.json`, `repair-queue.json`,
`receipt-audit.md`, artifact verification, source release confidence, or a
fixture-pinned 0.3.0 card-correctness gap, it waits.

## Done Criteria

The 0.3.0 lane is ready for release when:

- source `first-pr` emits the full review kit:
  `review-kit.json`, `cards.json`, `pr-summary.md`, `github-summary.md`,
  `cards.sarif`, `comment-plan.json`, `witness-plan.md`,
  `receipt-audit.md`, `lsp.json`, and `repair-queue.json`.
- artifact verification checks the full review kit.
- `pr-summary.md` is useful as the reviewer cockpit.
- `github-summary.md` stays bounded and points to artifacts.
- `comment-plan.json` has selected and not-selected rationale.
- `repair-queue.json` references valid cards and preserves agent boundaries.
- `receipt-audit.md` is metadata-only and emitted by `first-pr`.
- `unsafe-review explain <card-id>` and
  `unsafe-review context <card-id> --json` are readable, scoped, and
  non-overclaiming.
- `unsafe-review support` says all surfaces remain advisory/experimental.
- no-card output is honest and does not say or imply "all clear".
- bundle verification rejects missing artifacts and safety/proof wording.
- release notes describe the advisory review cockpit, not proof, policy
  authority, or Bun findings.

## Post-release Bun Use

Bun investigation is a post-release use case. The release provides the advisory
review kit; any Bun finding still requires manual proof of a safe API path,
local prevention, and a regression or witness appropriate to that finding.

Do not open Bun PRs for missing comments only, review gaps without witnessed
failure, broad refactors, or tool-pitch language.
