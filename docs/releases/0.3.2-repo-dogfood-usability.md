# 0.3.2 Repo Dogfood Usability Patch

`unsafe-review` 0.3.2 is a repo-dogfood usability patch. It publishes the
history-preserved source import of the swarm repo-mode dogfood batch.

The release goal is narrow:

- make `unsafe-review repo` usable on larger Rust repositories,
- preserve useful partial/status artifacts for long repo scans,
- include direct source locations in Markdown reports,
- accept manually discovered advisory candidates as first-class artifacts,
- keep related-sink clustering and JS getter reentry detection advisory,
- preserve the existing no-witness-execution, no-commenting, no-source-editing,
  and no-default-blocking boundaries.

## Highlights

- `unsafe-review repo --help`, `repo -h`, and `repo help` now show repo-specific
  behavior, large-repo guidance, output semantics, and trust-boundary text.
- Repo discovery supports `--include`, `--exclude`, `--respect-gitignore`,
  `--no-respect-gitignore`, `--max-files`, and `--list-files`.
- Repo scans support cooperative `--timeout-seconds` bounds; timeout exits keep
  incomplete status and the latest partial report when completed files exist.
- Repo scans use conservative large-repo default ignores for generated,
  vendored, build, and tool-output directories.
- Markdown repo reports include a compact `Location` column with `path:line`.
- Repo scans can write `<out>.status.json`; `--out` writes through
  `<out>.partial` and renames only after successful render.
- Manual candidate JSON can be imported and projected through explain, context,
  and witness-plan surfaces while remaining labeled as manual/advisory.
- `first-pr` writes `manual-candidates.json` when manual candidates are present
  and surfaces copy-only manual explain/context/witness-plan handoff commands in
  terminal output and `review-kit.json`.
- Repo Markdown includes a report-only related-sink cluster section.
- The scanner includes an advisory Bun/JSC-oriented heuristic for JS-backed
  descriptor capture, possible JS reentry, and later byte materialization.

## Versioning

The public crates move together for this patch:

```text
unsafe-review-core 0.3.1 -> 0.3.2
unsafe-review-cli  0.3.1 -> 0.3.2
unsafe-review      0.3.1 -> 0.3.2
```

This follows the synchronized public crate release pattern used by earlier
patch releases.

## Preconditions

Before publication:

- source `main` must contain source PR
  <https://github.com/EffortlessMetrics/unsafe-review/pull/508>,
- source merge commit `b399ece0` must be reachable from source `main`,
- imported swarm commit `edbb6da4` must be reachable from source history,
- no `0.3.1` release tag is rewritten,
- no `0.3.2` crate is published before dry-runs and package inspection pass.

## Non-goals

0.3.2 is not:

- memory-safety proof,
- UB-free status,
- Miri-clean status,
- site-execution proof,
- calibrated precision/recall,
- policy readiness,
- witness execution,
- automatic commenting,
- automatic source editing,
- default blocking policy.

## Trust Boundary

`unsafe-review` 0.3.2 reports static unsafe-review evidence and manually
imported advisory candidates. It does not prove memory safety, UB-free status,
Miri-clean status, site execution, witness success, calibrated precision/recall,
or policy readiness.

It does not execute witnesses, post comments, edit source, or enforce blocking
policy by default.
