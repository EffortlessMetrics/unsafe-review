# Explanation

Explanation pages answer why the project works the way it does. They provide
context, tradeoffs, and history; they are not the canonical place for schemas or
step-by-step PR recipes.

## Index

| Question | Read |
|---|---|
| What problem is `unsafe-review` trying to solve? | [`docs/MISSION.md`](../MISSION.md), [`docs/VISION.md`](../VISION.md) |
| How is the system shaped? | [`docs/ARCHITECTURE.md`](../ARCHITECTURE.md) |
| What is the release direction? | [`docs/ROADMAP.md`](../ROADMAP.md) |
| Why does a workstream exist? | [`docs/proposals/`](../proposals/) |
| Why was an architecture choice made? | [`docs/adr/`](../adr/) |

## Explanation rules

- Explain constraints, tradeoffs, and consequences.
- Link to specs when the reader needs the exact contract.
- Link to how-to guides when the reader needs commands to run.
- Do not duplicate policy ledgers or schemas here.
